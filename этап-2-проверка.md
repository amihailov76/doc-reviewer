## Шаги проверки

### Шаг 1. Обновите файлы проекта

Скачайте и разложите файлы согласно структуре выше.

### Шаг 2. Перезапустите бэкенд

В первом окне cmd (папка `C:\Projects\doc-reviewer`):

```
py -3.11 run_dev.py
```

Ожидаемый вывод — без ошибок импорта:
```
INFO:     Uvicorn running on http://0.0.0.0:8000
INFO:     Application startup complete.
```

### Шаг 3. Перезапустите фронтенд

Во втором окне cmd (папка `C:\Projects\doc-reviewer\frontend`):

```
npm run dev
```

### Шаг 4. Проверьте новые эндпойнты в Swagger

Откройте `http://localhost:8000/docs` и убедитесь, что в разделе **documents** появились эндпойнты:

- `POST /api/documents/upload`
- `POST /api/documents/upload/replace/{document_id}`
- `GET /api/documents/`
- `GET /api/documents/{document_id}/structure`
- `PATCH /api/documents/{document_id}/type`
- `DELETE /api/documents/{document_id}`

### Шаг 5. Загрузите документ через UI

1. Откройте `http://localhost:5173/document`
2. Перетащите файл в зону загрузки или нажмите и выберите файл на диске
3. Во время загрузки должен появиться спиннер с текстом «Загрузка и парсинг…»
4. После успешной загрузки браузер автоматически переходит на `/results/{id}`

**Тест с неподдерживаемым форматом:** попробуйте загрузить `.xlsx` или `.png` — должно появиться сообщение об ошибке под зоной загрузки, переход не происходит.

### Шаг 6. Проверьте дерево структуры

На странице результатов (`/results/{id}`) должны отображаться:

- Шапка с именем файла, типом (PDF/DOCX/MD/TXT) и количеством разделов
- Выпадающий список «Тип документа» с 5 вариантами
- Дерево разделов с заголовками документа

Проверьте поведение дерева:
- Разделы первых двух уровней открыты по умолчанию
- Узлы с дочерними элементами имеют стрелку `▸` / `▾`, клик сворачивает/разворачивает
- Узлы без детей отображают точку `·`
- У разделов PDF отображается номер страницы в виде бейджа `стр. N`

### Шаг 7. Проверьте выбор типа документа

1. На странице результатов выберите тип из выпадающего списка
2. Обновите страницу — тип должен сохраниться (проверяется через `GET /api/documents/{id}/structure` в Swagger)

До выбора типа должна отображаться подсказка: `⚠️ Укажите тип для корректной оценки`

### Шаг 8. Проверьте диалог при повторной загрузке

1. Загрузите тот же файл ещё раз (страница «Документ»)
2. Должен появиться модальный диалог с текстом о том, что файл уже существует
3. Нажмите **Отмена** — диалог закрывается, файл не заменяется, вы остаётесь на странице
4. Загрузите файл снова и нажмите **Заменить** — старый документ удаляется, новый загружается, переход на страницу результатов

### Шаг 9. Проверьте каждый формат

Загрузите по одному файлу каждого типа и убедитесь, что дерево не пустое:

| Формат | На что обращать внимание |
|---|---|
| **DOCX** | Заголовки должны соответствовать стилям `Heading 1–4` в документе |
| **PDF** | Заголовки определяются по размеру шрифта; при слабой иерархии шрифтов дерево может быть плоским — это нормально |
| **MD** | Заголовки по символам `#`; уровни соответствуют количеству `#` |
| **TXT** | Эвристика: строки полностью в верхнем регистре или заканчивающиеся на `:` из ≤8 слов считаются заголовками |

## Устранение проблем

| Проблема | Вероятная причина | Решение |
|---|---|---|
| Бэкенд падает с `ModuleNotFoundError: fitz` | PyMuPDF не установлен | `py -3.11 -m pip install PyMuPDF==1.24.3` |
| Бэкенд падает с `ModuleNotFoundError: docx` | python-docx не установлен | `py -3.11 -m pip install python-docx==1.1.0` |
| После загрузки файла нет перехода на результаты | Ошибка парсинга | Откройте консоль браузера (F12) и вкладку «Сеть» — найдите запрос к `/api/documents/upload` и посмотрите ответ |
| Дерево пустое для DOCX | В документе нет стилей `Heading` | Откройте документ в Word и убедитесь, что заголовки оформлены стилями, а не просто крупным шрифтом |
| Дерево пустое для PDF | Документ является сканом (изображение) | PyMuPDF не умеет распознавать текст из сканов — нужен OCR, это за рамками текущего этапа |
| Страница `/results` не открывается после загрузки | Роутер в `App.jsx` не обновлён | Убедитесь, что заменили `App.jsx` на новую версию с маршрутом `/results/:id` |
| CSS-стили не применяются | Не создана папка `styles` или импорты не совпадают | Убедитесь, что папка `frontend/src/styles/` создана и все три CSS-файла в ней есть |

## Выявленные и решённые проблемы

### 1. Предупреждение Pydantic: `Field "model_id" has conflict with protected namespace`

**Где:** вывод бэкенда при старте, файл `backend/routers/config_router.py`.

**Причина:** Pydantic резервирует префикс `model_` для внутренних нужд, поле `model_id` попадает под это ограничение.

**Решение:** в класс `SetActiveModelRequest` добавлена строка конфигурации:
```python
class SetActiveModelRequest(BaseModel):
    model_config = {"protected_namespaces": ()}
    model_id: str
```

### 2. Предупреждения в консоли браузера от React Router и Vite

**Сообщения:**
- `The CJS build of Vite's Node API is deprecated`
- `React Router Future Flag Warning: v7_startTransition`
- `React Router Future Flag Warning: v7_relativeSplatPath`

**Причина:** библиотеки сообщают о будущих изменениях в следующих мажорных версиях.

**Решение:** не требуется. Предупреждения косметические, на работу приложения не влияют.

### 3. Страница загрузки не обновилась после замены файлов

**Причина:** браузер закешировал старую версию `DocumentPage.jsx` (заглушку от этапа 1).

**Решение:** после замены файлов всегда выполнять жёсткую перезагрузку браузера — **Ctrl+Shift+R**.
