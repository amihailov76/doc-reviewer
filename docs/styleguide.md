# Стайлгайд технической документации

Правила написания и редактуры технических текстов, принятые в команде.
Этот файл читается при каждом обновлении пользовательского руководства (`docs/doc-reviewer-guide.pdf`).

---

## 1. Главное про тексты

### Задавайте вопросы к своему тексту

Хороший технический текст — полный, с максимумом контекста для выполнения сценария. При написании инструкции задавайте вопросы на каждом этапе:

| Этап сценария | Вопросы для контекста |
|---|---|
| **Начало** | Зачем нужно действие? Когда его выполнять? Какие предварительные условия? Какие ограничения? |
| **Описание процесса** | Сценарий сложный? Сначала описать воркфлоу. Сценарий простой? Можно описывать конкретные действия. Значения всех параметров понятны? Что учесть в процессе, действие влияет как-то на другие функции продукта? |
| **Конец** | Как (как долго) будут применяться изменения? Как будет при этом вести себя продукт? Какие результаты? Как они влияют на дальнейшую работу? Что делать дальше? |
| **Траблшутинг** | Что делать, если что-то пошло не так? Как откатить изменения? |

### Подобное описывается подобно

Подавайте единообразно все элементы документа:
- применяйте единую терминологию: начинайте описание нового продукта и создание документа с глоссария;
- используйте устойчивые словосочетания, одни и те же грамматические формы в подобных ситуациях;
- описывайте единообразно однородные объекты, процедуры, стандартные действия;
- стройте однотипные разделы единообразно с точки зрения организации информации;
- оформляйте единообразно одинаковые элементы текста (названия элементов интерфейса, таблиц, иллюстраций).

Описание однородных объектов, функций, возможностей, действий для большей наглядности оформляйте в виде списков.

### Кратко или подробно — вам решать

Тексты должны быть максимально краткими и простыми без потери смысла. Не экономьте слова там, где они действительно нужны. Текст должен дать весь нужный контекст для выполнения задачи, не перегружая пользователя избыточной информацией.

### Простые, нейтральные тексты

Пишите простые формулировки и предложения — тексты читают пользователи с разным техническим и культурным бэкграундом.

- Избегайте эмоциональных и категоричных высказываний.
- Применяйте восклицательный знак только для сценариев, связанных с серьёзным нарушением работы программы, угрозой для безопасности, потерей времени и данных.
- Не пишите «пожалуйста», не благодарите пользователя и не извиняйтесь перед ним.
- Обращайтесь к пользователю на **«вы»** (строчная).
- Не используйте метафоры и отсылки к локальной и массовой культуре.

### Термины, сленг и аббревиатуры

- Последовательно используйте термины; контролируйте единообразие терминологии.
- Сленг, устоявшийся в индустрии и понятный целевой аудитории, допустим — обсуждайте с редакторами и командой продукта.
- Не используйте слова разговорной речи.
- Если есть сомнения, поймёт ли аудитория сленг или аббревиатуру — расшифровывайте.

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Значения по умолчанию | Дефолтные значения |

## 2. Как строить абзацы и предложения

### Одна главная тема в абзаце

- Посвящайте абзац одной главной теме. Заявляйте тему в первом предложении.
- Начинайте предложения с уже известной информации, завершайте новой.
- В первом предложении абзаца не заменяйте местоимением существительное из предыдущего абзаца.
- В первом предложении раздела не заменяйте местоимением существительное, упомянутое в заголовке раздела.

### Одна тема в предложении

Разбивайте предложения с «ложными однородными членами»:

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Активировать программу вы можете с помощью файла ключа. Файл вам предоставляет Positive Technologies. | Файл ключа предоставляется вам Positive Technologies и позволяет активировать программу |

Разбивайте сложноподчинённые предложения, в которых придаточное содержит независимую тему:

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Отчёт содержит информацию о результатах выполненных операций. Из отчёта вы можете узнать о работе компонентов программы | Отчёт содержит информацию о результатах выполненных операций, благодаря чему вы можете узнать о работе компонентов программы |

### Действующие лица: пользователь и программа

Вводите в предложение действующих лиц — пользователя и программу. Стройте изложение на их взаимодействии. Обращайте безличные конструкции в личные, страдательный залог — в действительный.

| ✅ Хорошо | ❌ Плохо |
|---|---|
| При обнаружении уязвимости программа отправляет сообщение в Telegram ответственному специалисту | При обнаружении уязвимости отправляется сообщение в Telegram ответственному специалисту |

### Прямой порядок слов

Используйте прямой порядок слов: **подлежащее + сказуемое + второстепенные члены**. Не разделяйте подлежащее и сказуемое другими членами предложения.

Исключение: инверсия допустима только для связи предложений в абзаце (например: «…вкладка **Расписание**. На этой вкладке вы можете…»).

### Простое глагольное сказуемое

Заменяйте сказуемые с отглагольным существительным на простые глагольные сказуемые:

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Программа проверяет инфраструктуру организации | Программа производит проверку инфраструктуры организации |

Исключение: оставляйте отглагольные существительные там, где они делают текст кратким.

## 3. Как выбирать слова

### Утвердительные конструкции

Используйте утвердительные конструкции — их легче воспринимать. Выбирайте вариант без «не»:

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Укажите, какие записи нужно сохранить | Укажите, какие записи не нужно удалять |

### Пустые слова — удалять или заменять

Удаляйте или заменяйте слова, несущие мало информации: «на самом деле», «различный», «определённый», «непосредственный», «соответствующий», «тот или иной».

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Доступны три конфигурации системы | Доступны три возможных конфигурации системы |
| По ссылке **События** откройте окно с отчётом о событиях | По ссылке **События** откройте соответствующее окно с отчётом |

### Лишнее в контексте — удалять

Удаляйте слова, лишние в контексте: без них смысл такой же.

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Программа проверяет… | В процессе работы программа проверяет… |
| Перед началом сканирования нужно настроить получение данных | Для корректной работы программы перед началом сканирования нужно настроить получение данных |

### Тавтологии — убирать

Удаляйте слова, повторяющие по смыслу другие слова:

| ✅ Хорошо | ❌ Плохо |
|---|---|
| …откроется страница с версией программы и лицензией | …откроется информационная страница со сведениями о текущей версии программы и применяемой лицензии |
| Вы можете… | При необходимости вы можете… |
| Создайте папку… | Создайте новую папку… |
| В любой момент… | В любой момент времени… |

### Оговорки — заменять конкретикой

Избегайте слов, выражающих неуверенность: «в большинстве случаев», «в некоторых случаях», «фактически», «часто». Заменяйте оговорки конкретной информацией — у пользователя не должно оставаться вопросов о его частном случае.

### Усиления — не злоупотреблять

Не злоупотребляйте словами: «очень», «всегда», «только», «обязательно», «точно», «абсолютно», «несомненно», «конечно», «никак». Используйте их только там, где они уместны (например, в блоках «Внимание» для предупреждения о негативных последствиях).

### Авторская речь — не применять

Не используйте слова и словосочетания, выражающие намерения и рассуждения автора: «рассмотрим», «вам предлагается», «стоит отметить», «необходимо отметить, что», «ясно, что».

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Система состоит из трёх компонентов: | Рассмотрим, из каких компонентов состоит система… |

## 4. Нейтральный тон текста

### Без оценочных суждений

Не делайте оценочных суждений в тексте. Избегайте слов: «хороший», «плохой», «большой», «мелкий», «важный», «полный».

Не используйте слово **«полный»** в отношении полноты информации в документах — такие формулировки несут юридические риски.

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Это руководство содержит информацию о том, как использовать программу | Это руководство содержит полный набор сведений, необходимых для использования программы |

### Этикет

- Не принижайте уровень квалификации пользователя.
- Не заменяйте слово «пользователь» местоимением «он».
- Исключайте политические, половые, этнические, конфессиональные, социальные и иные характеристики, если только они не предопределены назначением программы.
- Исключайте эмоциональные, шутливые и иронические высказывания.

### Пишите «вы можете»

Выражайте возможность словами **вы можете**. Не пишите «допускается» — у этого слова оттенок разрешения.

### Пишите «вам нужно»

Описывая обязательное действие, употребляйте: **нужно**, **требуется**, **необходимо**. Не употребляйте слова с оттенком обязанности: «вы должны», «вам следует».

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Вам нужно выполнить следующие действия: | Вам следует выполнить следующие действия: |

### Ограничения — не как запрет

Не формулируйте ограничение возможностей пользователя как запрет:

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Пользователь с ролью «оператор» не может добавлять пользователей | Пользователю с ролью «оператор» запрещено добавлять пользователей |

### Не очеловечивайте программу

Не пишите формулировки, уподобляющие программу человеку: «продукт умеет», «система не знает», «сервер думает», «программа приняла это во внимание». Допустимы «человеческие» действия, если они соответствуют общепринятой терминологии: «программа опрашивает сеть», «программа анализирует события».

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Программа оценивает уровень опасности инцидента и даёт инженеру рекомендации | Программа смотрит на уровень опасности инцидента и говорит инженеру как вести расследование |

## 5. Как писать инструкции

### Анализ задачи перед написанием

Перед написанием инструкции определите:

- **Цель пользователя** — что именно пользователь хочет достичь, а не какие кнопки нажать.
- **Сложность задачи** — от этого зависит структура инструкции:

| Тип задачи | Описание |
|---|---|
| **Сквозной сценарий** | Пользователь проходит через несколько функциональных блоков; начинается с создания сущности и заканчивается получением результата |
| **Составная задача** | Несколько связанных локальных задач; результат зависит от их совместного выполнения |
| **Локальная задача** | Одна операция в одном функциональном блоке |
| **Операция** | Одно атомарное действие; описывается как шаг внутри инструкции, а не отдельной инструкцией |

- **Декомпозиция** — разбейте сложную задачу на локальные, прежде чем писать инструкцию.

### Общие требования к инструкции

- **Одна инструкция — одна задача.** Не объединяйте несвязанные задачи в одну инструкцию.
- **Не пишите скрытых инструкций.** Если пользователь должен что-то сделать — вынесите это в явный шаг, а не описывайте вскользь в тексте.
- **Группируйте инструкции** по разделам, если задачи связаны одной темой или одним объектом.

### Заголовок инструкции

- Формулируйте заголовок как отглагольное существительное: **«Создание проекта»**, **«Удаление документа»**.
- Заголовок отражает цель пользователя, а не интерфейсный элемент.
- Не привязывайте заголовок к интерфейсу: не «Использование кнопки Оценить», а «Оценка документа».

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Настройка модели | Использование страницы настроек |
| Создание проекта | Нажатие кнопки «Создать» |

### Вводный текст

Вводный текст предшествует титульной фразе и шагам. Используйте его, когда нужно:

- объяснить, когда и зачем выполнять задачу;
- перечислить предварительные условия;
- предупредить о негативных последствиях.

**Не делайте во вводном тексте:**

- Не повторяйте заголовок и титульную фразу другими словами.
- Не описывайте действия — действия только в шагах.
- Не перегружайте текст теорией; если нужен контекст — дайте ссылку.

### Титульная фраза

Титульная фраза открывает инструкцию перед шагами.

- Начинается с **«Чтобы…»** и описывает цель.
- Соответствует заголовку инструкции по смыслу.
- Завершается двоеточием.

**Пример:** «Чтобы оценить документ по критериям:»

### Шаг инструкции

Каждый шаг — одно действие пользователя.

- Начинайте шаг с **глагола в повелительном наклонении**: «Нажмите», «Выберите», «Введите».
- Пишите **одно действие на шаг** — не объединяйте несколько действий через запятую или союз «и».
- **Начинайте с указания места** в интерфейсе, если оно не очевидно: «На панели инструментов нажмите…», «В разделе **Настройки** откройте…».
- **Не пропускайте «очевидные» шаги** — то, что кажется очевидным автору, может быть неочевидным пользователю.
- **Не выделяйте ожидание в отдельный шаг** («Дождитесь загрузки» — не шаг инструкции, это промежуточный результат).

| ✅ Хорошо | ❌ Плохо |
|---|---|
| Нажмите **Оценить**. | Нажмите **Оценить** и дождитесь результата. |
| В разделе **Проекты** нажмите **Создать проект**. | Создайте проект. |

### Условный шаг

Если шаг выполняется не всегда, а при определённом условии:

- Начинайте с **«Если…»**: «Если нужно добавить документ в проект, выберите проект из списка.»
- Не включайте условный шаг в основную нумерацию как обязательный.

### Пояснение к шагу

Пояснение — дополнительная информация под шагом (не новый шаг).

- Используйте для уточнения значений параметров, примеров, ограничений.
- Не повторяйте в пояснении то, что уже сказано в шаге.
- Пояснение не заменяет шаг — действие должно быть в самом шаге.

### Промежуточный результат

Промежуточный результат описывает состояние интерфейса после группы шагов.

- Используйте, когда после нескольких шагов интерфейс заметно меняется и пользователю нужно убедиться, что он на правильном пути.
- **Не описывайте каждую смену интерфейса** — только значимые изменения.
- Пишите в настоящем времени: «Откроется окно **Настройки проекта**.»

### Результат выполнения

Результат выполнения — финальный абзац инструкции.

- Пишите в **прошедшем времени**: «Проект создан.», «Документ добавлен в проект.»
- Результат соответствует цели из титульной фразы — закрывает «Чтобы…».
- При необходимости указывайте, что происходит дальше или как проверить результат.

### Структуры инструкций

Выбирайте структуру в зависимости от сложности задачи:

**Инструкция из одного шага**

Применяется, когда задача решается одним действием.

```
Заголовок

Вводный текст (если нужен).

Чтобы [цель]:
[Один шаг.]

[Результат.]
```

**Последовательная инструкция**

Применяется для задач с чётко определённым порядком шагов. Не более 9 шагов — если больше, декомпозируйте на подзадачи.

```
Заголовок

Чтобы [цель]:
1. Шаг первый.
2. Шаг второй.
3. Шаг третий.

[Результат.]
```

**Инструкция с условными шагами**

Применяется, когда часть шагов выполняется при определённых условиях.

```
Чтобы [цель]:
1. Обязательный шаг.
2. Если [условие], выполните действие.
3. Следующий обязательный шаг.

[Результат.]
```

**Инструкция с несколькими вариантами**

Применяется, когда одну задачу можно выполнить несколькими способами.

```
Чтобы [цель], воспользуйтесь одним из способов.

**Способ 1. [Название]**
1. ...

**Способ 2. [Название]**
1. ...
```

**Инструкция с вложенными шагами**

Применяется, когда один из шагов требует уточнения в виде подшагов. Используйте умеренно — глубокая вложенность затрудняет восприятие.

```
1. Основной шаг:
   а. Подшаг.
   б. Подшаг.
2. Следующий основной шаг.
```

**Инструкция с развилкой**

Применяется, когда дальнейшие действия зависят от выбора пользователя. Развилка разделяет инструкцию на две ветки после определённого шага.

```
Чтобы [цель]:
1. Общий шаг.
2. Общий шаг.

Если [условие А]:
3. Шаг для варианта А.
4. Шаг для варианта А.

Если [условие Б]:
3. Шаг для варианта Б.
```
