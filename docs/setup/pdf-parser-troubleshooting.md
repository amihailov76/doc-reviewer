# Траблшутинг парсера PDF

Живой документ. Пополняется по мере обнаружения новых проблем при загрузке PDF с разными шаблонами оформления.

---

## Как устроен PDF-парсер

Парсер определяет заголовки разделов **по размеру шрифта** относительно медианного размера по всему документу:

- медианный размер шрифта = «основной текст»
- блок крупнее медианы на ≥15% → заголовок (уровень 1–4 в зависимости от соотношения)
- блок меньше порога → тело текущего раздела

Это работает для большинства PDF с явной иерархией шрифтов, но ломается при нестандартном оформлении.

---

## Известные проблемы и решения

### 1. Титульные фразы инструкций определяются как заголовки

**Симптом:** строки «► Чтобы установить роль:» создают отдельный раздел в дереве вместо того, чтобы быть частью тела родительского раздела. В результате контент инструкции «разъезжается» по двум записям — ни одна не набирает все три признака и не получает классификацию `instruction`.

**Причина:** в шаблоне документа фраза «Чтобы...» оформлена жирным или увеличенным шрифтом, что превышает порог определения заголовка.

**Решение:** добавлен список `_BODY_TEXT_PATTERNS` — паттерны строк, которые никогда не считаются заголовками вне зависимости от размера шрифта. Паттерн для «Чтобы»:
```python
r"^(?:[►▶•]\s*)?[Чч]тобы\s"
```

**Как диагностировать:** найти в дереве раздел с классификацией `possible`, открыть через `GET /api/instructions/{id}` — если `content` содержит только вводный абзац без шагов, а соседний раздел начинается с «Чтобы» и содержит нумерованные шаги — это данная проблема.

---

### 2. Навигационные блоки «В этом разделе» и «См. также» определяются как заголовки

**Симптом:** в разделах верхнего уровня появляются дочерние разделы с заголовком «В этом разделе» или «См. также», содержащие список ссылок на подразделы.

**Причина:** эти блоки оформлены увеличенным шрифтом как визуальные подзаголовки, но не являются структурными разделами документа.

**Решение:** добавлены паттерны в `_BODY_TEXT_PATTERNS`:
```python
r"|^[Вв]\s+этом\s+разделе"
r"|^[Сс]м\.?\s+также"
```

---

### 3. Список ссылок на подразделы (4.1, 4.2...) срабатывает как признак «нумерованные шаги»

**Симптом:** разделы-оглавления типа «4. Резервное копирование» получают классификацию `possible` или `instruction`, хотя содержат только список ссылок на подразделы, а не пошаговую инструкцию.

**Причина:** детектор (признак 3) находил строки «4.1 Создание резервной копии», «4.2 Восстановление» и считал их нумерованным списком шагов.

**Решение:** признак 3 теперь требует, чтобы нумерованный список **начинался с `1.` или `1)`**. Списки, начинающиеся с 4.1, 4.2 и т.п., не считаются шагами:
```python
return "1" in matches  # список должен начинаться с единицы
```

---

### 4. Строки маркированного списка с тире определяются как заголовки

**Симптом:** строки вида «— если синхронизация выполнена успешно» создают отдельные разделы в дереве.

**Причина:** в PDF каждый пункт маркированного списка с тире — отдельный текстовый блок, иногда с увеличенным шрифтом.

**Решение:** добавлен паттерн в `_BODY_TEXT_PATTERNS`:
```python
r"|^[\-\—\–]\s+\S"
```

---

### 5. Символьные шрифты (иконки) создают мусорные разделы

**Симптом:** в дереве появляются разделы с заголовком «□» или бессмысленным набором символов. Строки вида «□ — система работает корректно» разбиваются на части.

**Причина:** PDF использует символьные шрифты для иконок (✓, ⚠, кнопки интерфейса). PyMuPDF извлекает их как символы из **Private Use Area Unicode** (диапазон `\uE000–\uF8FF`), например `\uf7ef`, `\uf531`. Каждая иконка — отдельный текстовый спан или блок.

**Как выявить точные коды символов:**
```
py -3.11 -c "import fitz; doc = fitz.open('data/uploads/ВАШ_ФАЙЛ.pdf'); page = doc[N]; blocks = page.get_text('dict')['blocks']; [print(repr(b['lines'][0]['spans'][0]['text'][:30])) for b in blocks if b['type']==0 and b['lines']]"
```
Коды вида `'\uf7ef'` подтверждают PUA-символы.

**Решение:** добавлена функция `_clean_pua()` — удаляет все символы из диапазона `\uE000–\uF8FF` из текста блока перед обработкой. Блоки, состоящие только из таких символов, отбрасываются через `_is_garbage_block()`.

---

### 6. Иконки символьных шрифтов завышают определяемый уровень заголовка

**Симптом:** обычные строки описания интерфейса («и кнопки . В таблице доступны...») получают level 3 и создают отдельные разделы, хотя их текст имеет стандартный размер шрифта.

**Причина:** иконки в строке имеют размер шрифта 15pt, тогда как текст — 11pt. `block_max_size` вычислялся по всем спанам блока включая иконки, из-за чего блок получал завышенный уровень.

**Как диагностировать:** если медиана шрифта документа равна 11pt, а проблемные блоки показывают `size=15.0` — скорее всего это данная проблема. Проверить медиану:
```
py -3.11 -c "import sys; sys.path.insert(0, '.'); import fitz; from statistics import median; doc = fitz.open('data/uploads/ВАШ_ФАЙЛ.pdf'); sizes = [sp['size'] for page in doc for b in page.get_text('dict')['blocks'] if b['type']==0 for l in b['lines'] for sp in l['spans'] if sp['text'].strip()]; print('median:', round(median(sizes),2)); from collections import Counter; print('top:', Counter(round(s,1) for s in sizes).most_common(5))"
```

**Решение:** `block_max_size` теперь вычисляется только по спанам, текст которых содержит реальные символы после очистки PUA:
```python
span_clean = _clean_pua(span["text"]).strip()
if span_clean and span["size"] > block_max_size:
    block_max_size = span["size"]
```

## Как добавить новое исключение для заголовка

Если в документе встречается блок, который оформлен крупным шрифтом, но не является структурным заголовком раздела — добавьте паттерн в список `_BODY_TEXT_PATTERNS` в файле `backend/services/parser/pdf_parser.py`:

```python
_BODY_TEXT_PATTERNS = re.compile(
    r"^(?:[►▶•]\s*)?[Чч]тобы\s"
    r"|^\d+[\.\)]\s"
    r"|^[Вв]\s+этом\s+разделе"
    r"|^[Сс]м\.?\s+также"
    r"|^[\-\—\–]\s+\S"
    r"|^[\u25A1\u25AA\u25CF\u2022\u25B6]\s*"
    r"|^[\uE000-\uF8FF]"
    r"|^ВАШ_НОВЫЙ_ПАТТЕРН",   # ← добавьте сюда
    re.MULTILINE,
)
```

После изменения файла:
1. Перезапустите бэкенд
2. Загрузите документ повторно через «Заменить» — нужен полный перепарсинг
3. Переклассификация через `/reclassify` **не поможет** — она не перепарсирует файл, только переприменяет детектор к уже сохранённым разделам

## Когда переклассификация помогает, а когда нет

| Тип изменения | Достаточно `/reclassify` | Нужна перезагрузка файла |
|---|---|---|
| Изменена логика детектора (`detector.py`) | ✓ | |
| Изменены суффиксы девербативов | ✓ | |
| Изменены паттерны `_BODY_TEXT_PATTERNS` | | ✓ |
| Изменена логика `_is_garbage_block` | | ✓ |
| Изменён порог `HEADING_SIZE_THRESHOLD` | | ✓ |
| Изменена логика подсчёта `block_max_size` | | ✓ |
