## О библиотеке pymorphy2

На этом этапе в проект добавлен морфологический анализатор **pymorphy2** — он используется для определения отглагольных существительных в заголовках разделов (признак 1 классификации).

**Почему pymorphy2, а не список суффиксов:**
- pymorphy2 выполняет полный морфологический разбор слова и возвращает часть речи, граммемы и уверенность разбора
- суффиксный подход даёт ложные срабатывания на словах, которые оканчиваются похоже, но не являются девербативами
- для больших документов с сотнями разделов точность важнее скорости

**Как работает классификация:**
- Признак 1 (отглагольное существительное в заголовке): каждое слово длиннее 4 символов проверяется через pymorphy2 на тег NOUN + совпадение с продуктивными суффиксами девербативов (`-ение`, `-ание`, `-ация`, `-ировка`, `-овка`, `-тие` и др.)
- Признак 2 (фраза с «чтобы»): регулярное выражение ищет строки вида «Чтобы <глагол>…:»
- Признак 3 (нумерованные шаги): не менее двух строк, начинающихся с `1.`, `1)` и т.п.

**Результат классификации:**
- все три признака → `instruction` (синий бейдж в дереве)
- один или два признака → `possible` (жёлтый бейдж)
- ноль признаков → `non-instruction` (бейдж не отображается)

**Первый запуск после установки:** pymorphy2 загружает словари русского языка (~20 МБ) из пакета `pymorphy2-dicts-ru`. Загрузка происходит один раз при первом обращении к анализатору и занимает около 0.5 секунды. При последующих запусках словари берутся из кеша.

## Шаги проверки

### Шаг 1. Установите новые зависимости

В командной строке (папка `C:\Projects\doc-reviewer`):

```
py -3.11 -m pip install pymorphy2==0.9.1 pymorphy2-dicts-ru==2.4.417127.4579844
```

Ожидаемый результат в конце: `Successfully installed pymorphy2-... pymorphy2-dicts-ru-...`

### Шаг 2. Разложите файлы по структуре

Скачайте и разложите файлы согласно таблице выше.

### Шаг 3. Перезапустите бэкенд

```
py -3.11 run_dev.py
```

Ожидаемый вывод — без ошибок импорта:
```
INFO:     Uvicorn running on http://0.0.0.0:8000
INFO:     Application startup complete.
```

### Шаг 4. Проверьте новые эндпойнты в Swagger

Откройте `http://localhost:8000/docs` и убедитесь, что появились:

В разделе **instructions:**
- `GET /api/instructions/possible`
- `GET /api/instructions/{instruction_id}`
- `PATCH /api/instructions/{instruction_id}/include`

В разделе **documents:**
- `POST /api/documents/{document_id}/reclassify`

### Шаг 5. Загрузите или замените документ

Перейдите на страницу «Документ» (`http://localhost:5173/document`) и загрузите файл.

> Если документ уже был загружен на этапе 2, загрузите его повторно через диалог «Заменить» — это запустит классификацию с новым детектором.

### Шаг 6. Проверьте счётчики классификации

На странице результатов в шапке документа должны появиться три счётчика:

```
✓ N инструкций    ? N возможных    — N прочих
```

Все три числа в сумме должны равняться общему количеству разделов документа.

### Шаг 7. Проверьте бейджи в дереве

В дереве структуры проверьте:
- Разделы с заголовками типа «Добавление записи», «Настройка источника», «Создание правила» — должны получить синий бейдж **инструкция**
- Разделы с одним-двумя признаками — жёлтый бейдж **возможная**
- Разделы без признаков — без бейджа

### Шаг 8. Проверьте включение/исключение из оценки

У каждого раздела с бейджем «инструкция» или «возможная» справа должна быть кнопка `✓`.

1. Нажмите `✓` на любом разделе — кнопка должна смениться на `✗`, заголовок раздела зачеркнётся и станет полупрозрачным
2. Нажмите `✗` — раздел снова включится, зачёркивание исчезнет
3. Перезагрузите страницу — состояние включения/исключения должно сохраниться (данные хранятся в БД)

### Шаг 9. Проверьте повторную классификацию через Swagger

Откройте `http://localhost:8000/docs`, найдите эндпойнт `POST /api/documents/{document_id}/reclassify`, введите `id` загруженного документа и выполните запрос.

Ожидаемый ответ:
```json
{
  "ok": true,
  "classification": {
    "instruction": N,
    "possible": N,
    "non-instruction": N
  }
}
```

## Устранение проблем

| Проблема | Вероятная причина | Решение |
|---|---|---|
| Бэкенд падает с `ModuleNotFoundError: pymorphy2` | Библиотека не установлена | Выполните `py -3.11 -m pip install pymorphy2==0.9.1 pymorphy2-dicts-ru==2.4.417127.4579844` |

## Выявленные и решённые проблемы

### 1. pymorphy2 несовместима с Python 3.11

**Ошибка:** `AttributeError: module 'inspect' has no attribute 'getargspec'`

**Причина:** `pymorphy2==0.9.1` использует `inspect.getargspec`, удалённую в Python 3.11.

**Решение:** заменить на `pymorphy3==2.0.2` — совместимый форк. В `detector.py` импорт алиасируется:
```python
import pymorphy3 as pymorphy2
```
В `requirements.txt` заменить строки pymorphy2 на:
```
pymorphy3==2.0.2
```

### 2. Титульная фраза «► Чтобы...» разбивала инструкцию на два раздела

**Симптом:** 0 инструкций, все разделы — `possible`. Раздел «Установка роли» содержал только вводный абзац, а «► Чтобы установить роль:» с нумерованными шагами — отдельную запись.

**Причина:** PDF-парсер определял строку с «Чтобы» как заголовок из-за увеличенного шрифта.

**Решение:** добавлен список `_BODY_TEXT_PATTERNS` в `pdf_parser.py`. Подробнее — в `pdf-parser-troubleshooting.md`.

### 3. Навигационные блоки, тире-списки, иконки создавали мусорные разделы

**Симптомы:** в дереве появлялись разделы «В этом разделе», «См. также», «□ — текст», одиночные тире.

**Причина:** PDF-парсер определял их как заголовки из-за размера или жирности шрифта. Иконки из символьных шрифтов (PUA Unicode `\uE000–\uF8FF`) завышали `block_max_size` блока.

**Решение:** расширен `_BODY_TEXT_PATTERNS`, добавлены `_clean_pua()` и `_is_garbage_block()`, исправлен подсчёт `block_max_size`. Подробнее — в `pdf-parser-troubleshooting.md`.

### 4. Оглавления вида «4.1, 4.2...» срабатывали как нумерованные шаги

**Симптом:** разделы-оглавления получали классификацию `possible` из-за ложного срабатывания признака 3.

**Решение:** признак 3 теперь требует наличия `1.` или `1)` в начале списка.
