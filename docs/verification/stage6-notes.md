# Заметки этапа 6 — к финальной документации

## Архитектурные решения

### Критерии оценки — автоматическая синхронизация с criteria.md

**Цепочка:** `criteria.md` → `/api/config/criteria` → `DocumentContext` → `FeedbackPanel`

- Бэкенд: `config_router.py`, функция `_parse_criteria()` — парсит заголовки вида `### 1.1 Название\nОписание`
- Фронтенд: `DocumentContext.jsx` загружает критерии один раз при старте через `useEffect`
- Компонент `FeedbackPanel` получает `criteriaLabels` из контекста

**Важно для сопровождения:** при изменении `criteria.md` нужно перезапустить только бэкенд. Фронтенд пересобирать не нужно.

**Использование на UI:** tooltip на бейджах критериев (1.1, 1.2...) + расшифровка названия в заголовке каждой рекомендации.

---

### Повторная загрузка документа — без диалога подтверждения

Многократная загрузка одного и того же файла — основной сценарий работы. Диалог «файл уже загружен» убран, замена происходит автоматически.

## Исправленные баги

### Layout EvaluationPage не растягивался на всю ширину

**Причина:** `.main-content { flex: 1 }` без `min-width: 0`. В flex-контейнере дочерний элемент не сжимается ниже размера своего содержимого без явного `min-width: 0`. Длинные заголовки разделов в дереве распирали контейнер вширь.

**Исправление:** `App.css` → `.main-content` → добавлен `min-width: 0`.

**Почему Снимки работали, а Оценка — нет:** на странице Снимков нет широких элементов-распорок, поэтому баг не проявлялся.

---

### PDF-парсер пропускал разделы верхнего уровня с нумерацией

**Симптом:** разделы `2. О PT Knockin`, `3. Вход`, `6. Управление проверкой`, `7. Запуск атаки` и т.д. полностью отсутствовали в структуре документа.

**Причина:** в `_BODY_TEXT_PATTERNS` был паттерн `r"^\d+[\.]\s"`, задуманный для нумерованных шагов инструкций (`1. Откройте сайт`). Но он матчил и заголовки разделов (`2. О PT Knockin`), принудительно понижая их до уровня тела текста.

**Почему `1.1. Название` не ломался:** паттерн ищет `\d+\.\s` — после `1.` ожидает пробел, но следует вторая точка. Случайная защита.

**Почему паттерн был лишним:** нумерованные шаги инструкций написаны обычным шрифтом. Проверка паттернов применяется только к блокам с крупным шрифтом (`level > 0`). Шаги до неё не доходят — отсеиваются раньше по размеру.

**Исправление:** строка `r"|^\d+[\.]\s"` удалена из `_BODY_TEXT_PATTERNS` в `pdf_parser.py`.

## Этап 7: API-ключи и управление моделями

### Архитектурные решения

**Ключ привязан к модели, не к провайдеру.**
Каждая модель хранит свой `api_key` в таблице `models`. Отдельная таблица `ApiKey` упразднена.
Обоснование: один пользователь может иметь несколько моделей одного провайдера с разными ключами.

**Модели переехали из models.yml в БД.**
При первом запуске `_migrate_models_from_yml()` переносит модели из `models.yml` в таблицу `models`. Дальнейшее управление — только через UI.

**API-ключ никогда не возвращается на фронтенд.**
Эндпойнт отдаёт только `has_key: true/false`. Сам ключ доступен только бэкенду.

**Приоритет ключа в evaluator:** ключ модели из БД → переменная окружения.

### Исправленные баги

**Дублирование классов в database.py.**
Причина: `str_replace` добавил новый класс `Model` в начало файла, затем `cat >>` дописал функцию миграции в конец — вместе со старыми классами которые там уже были. SQLAlchemy упало с `InvalidRequestError: Table already defined`.
Исправление: файл переписан целиком.

### Порядок сборки

1. Убедиться что установлен Python 3.11 и Node.js
2. Запустить из корня проекта:
   ```

   ```
3. Скрипт автоматически:
   - Собирает фронтенд (`npm run build` → `frontend/dist/`)
   - Устанавливает зависимости Python
   - Запускает PyInstaller

Первая сборка занимает 5–10 минут.

### Результат

```
dist\doc-reviewer.exe
```

### Поведение .exe при запуске

- Находит свободный порт автоматически
- Запускает бэкенд (FastAPI + uvicorn) на этом порту
- Через 2 секунды открывает браузер
- Фронтенд раздаётся как статика из встроенного `frontend/dist/`

### Расположение данных рядом с .exe

```
dist\
├── doc-reviewer.exe
├── data\              ← создаётся автоматически при первом запуске
│   └── db.sqlite      ← БД с моделями, снимками, оценками
├── criteria.md        ← критерии оценки (редактируются вручную)
└── models.yml         ← начальный список моделей (seed при первом запуске)
```

### Важные замечания

- `criteria.md` и `models.yml` упакованы в `.exe`, но при необходимости их можно положить рядом с `.exe` — бэкенд найдёт их там
- При обновлении `criteria.md` достаточно заменить файл рядом с `.exe` — пересборка не нужна
- `db.sqlite` не входит в `.exe` — данные сохраняются между запусками

## Управление критериями оценки

### Архитектурные решения

**Критерии хранятся в БД, не в файле.**
Таблица `CriteriaSet` — id, name, content (полный Markdown), is_active, is_default.
При первом запуске `criteria.md` мигрирует в БД с флагом `is_default=True`.

**Нумерация критериев свободная** — паттерн парсера обновлён с `\d+\.\d+` до `\d+[\.\d]*`, что позволяет использовать 1.1, 1.4, 4.1 и т.д.

## Обработка символьных шрифтов (PUA) в PDF

### Проблема

В PDF-документации PT используются иконки интерфейса из символьных шрифтов (например, Mosaic). Эти иконки реализованы двумя способами:

1. **Текстовый спан с PUA-символом** (U+E000–U+F8FF) — PyMuPDF извлекает как текст в том же блоке что и окружающий текст. Пример: `изменение атаки (\uf146 → Изменить)`.

2. **Графический элемент поверх текста** — PyMuPDF не извлекает вообще. Пример: `нажатию [иконка не видна] в главном меню`. В распарсенном тексте остаётся пропуск: `Нажмите .`

### Решение для случая 1 — замена PUA на [иконка]

В `pdf_parser.py`, функция `_clean_pua`:
```python
# Было: удаление
return re.sub(r"[\uE000-\uF8FF]", "", text).strip()

# Стало: замена на метку
return re.sub(r"[\uE000-\uF8FF]+", "[иконка]", text).strip()
```

Несколько PUA-символов подряд схлопываются в одну метку.

**Важно:** `_clean_pua` вызывается ДО `_is_garbage_block`. Это намеренно — иначе одиночные PUA-блоки (len=1) отбрасывались бы фильтром `len <= 2` до замены.

### Решение для случая 2 — инструкция в системном промпте

Графические иконки парсером не восстанавливаются — это ограничение формата PDF. Решение на стороне LLM: в `SYSTEM_PROMPT` в `evaluator.py` добавлена явная инструкция:

> Иконки интерфейса могут быть обозначены меткой [иконка] или отсутствовать совсем. Не считай отсутствие или метку [иконка] ошибкой при оценке критерия 2.1 — оценивай только наличие названий кнопок, полей и пунктов меню в тексте.

### Диагностические скрипты

| Скрипт | Назначение |
|---|---|
| `debug_pua.py` | Находит PUA-символы в PDF и показывает шрифты |
| `debug_pua2.py` | Показывает все нестандартные символы с кодами |
| `debug_block.py` | Структура блоков содержащих PUA |
| `debug_all_blocks.py` | Все блоки на заданной странице |
| `debug_db.py` | Проверяет содержимое БД после загрузки |
| `debug_content.py` | Полный контент разделов из БД |

## Обработка символьных шрифтов (PDF-парсер)

### Проблема

Многие инструкции в PDF-документации PT используют иконки из символьных шрифтов (например, шрифт `Mosaic`, символ `U+F146`). При извлечении текста парсером иконки терялись — LLM видел пропуски в тексте (например `Нажмите .`) и расценивал их как ошибку документации (критерий 2.1).

### Два типа иконок в PDF

**Тип 1 — текстовый спан в PUA-диапазоне** (U+E000–U+F8FF).  
Иконка находится в том же блоке что и окружающий текст, в отдельном спане шрифта-иконки.  
Пример: `изменение атаки (\uf146 → Изменить)` — шрифт Mosaic, спан внутри блока.  
Решение: `_clean_pua()` в `pdf_parser.py` заменяет PUA-символы на `[иконка]`.

**Тип 2 — графический элемент** (inline image или vector shape).  
Иконка нарисована поверх текста как графика — PyMuPDF не видит её через `get_text("dict")` вообще.  
Пример: `открывающейся по нажатию [пусто] в главном меню` — иконка графическая, текстового представления нет.  
Это ограничение формата PDF, парсером не решается.

### Решения

**pdf_parser.py — замена PUA на метку:**
```python
# Было:
return re.sub(r"[\uE000-\uF8FF]", "", text).strip()
# Стало:
return re.sub(r"[\uE000-\uF8FF]+", "[иконка]", text).strip()
```
Несколько подряд идущих PUA-символов схлопываются в одну метку.

**evaluator.py — инструкция в системном промпте:**
Добавлен абзац: не считать ошибкой отсутствие иконки или метку `[иконка]` при оценке критерия 2.1 — оценивать только наличие текстовых названий кнопок, полей и меню.

### Диагностические скрипты

| Скрипт | Назначение |
|---|---|
| `debug_pua.py` | Находит PUA-символы в PDF, показывает шрифты |
| `debug_pua2.py` | Все нестандартные символы с кодами и категориями |
| `debug_block.py` | Структура блоков содержащих PUA |
| `debug_all_blocks.py` | Все блоки страницы подряд |
| `debug_db.py` | Проверяет наличие `[иконка]` в БД после загрузки |
| `debug_content.py` | Показывает контент разделов из БД |
