# doc-reviewer

Приложение для автоматической оценки полноты инструкций в технической документации. Загружает документ (PDF, DOCX, MD, TXT), автоматически выделяет разделы с инструкциями и оценивает каждый из них по настраиваемым критериям с помощью LLM.

Разработан для технических писателей продуктов Positive Technologies.

---

## Возможности

- 📁 **Проекты** — группировка документов одного продукта, автогенерируемый контекст продукта для улучшения точности оценки
- 📄 **Парсинг документов** — PDF, DOCX, Markdown, TXT
- 🌐 **Оценка веб-страниц** — загрузка инструкций по URL через headless-браузер (Playwright + Chromium); поддержка JS-сайтов и SPA; оптимизировано для веб-справки Positive Technologies
- 🔍 **Автоматическое определение инструкций** — морфологический анализ на русском языке (pymorphy3)
- 🤖 **Оценка через LLM** — поддержка OpenAI, Anthropic и любых OpenAI-совместимых API (включая локальные модели через Ollama)
- 📊 **Экспорт результатов** в XLS
- 📸 **Снимки и сравнение версий** — сохранение состояния оценки и сравнение между версиями документа
- ⚙️ **Управление моделями** — добавление и настройка LLM-моделей через интерфейс без редактирования файлов
- 📝 **Управление критериями** — редактирование и создание наборов критериев оценки через интерфейс
- 💾 **Локальное хранение** — SQLite, все данные на машине пользователя
- 📦 **Дистрибуция** — сборка в единый `.exe` через PyInstaller

## Быстрый старт

### Способ 1. Установка готового приложения (рекомендуется)

Подходит для использования без настройки среды разработки. Требуется Windows 10/11 (64-bit).

#### Шаг 1. Скачайте приложение

Перейдите на страницу [Releases](https://github.com/amihailov76/doc-reviewer/releases) и скачайте файл `doc-reviewer.exe` из последнего релиза (раздел **Assets**).

#### Шаг 2. Создайте папку для приложения

Создайте отдельную папку, например `C:\doc-reviewer\`, и переместите туда скачанный файл.

> **Почему важна отдельная папка:** при первом запуске рядом с `.exe` автоматически создаётся папка `data\` с базой данных. Там будут храниться все ваши документы, оценки и настройки. Держать приложение в отдельной папке удобнее для резервного копирования и обновлений.

#### Шаг 3. Запустите приложение

Дважды щёлкните по `doc-reviewer.exe`. Программа автоматически откроет браузер с интерфейсом.

Если браузер не открылся в течение 5–10 секунд — откройте любой браузер вручную. Точный адрес отображается в заголовке вкладки браузера после старта приложения (например, `http://localhost:8765`).

> **Не закрывайте окно doc-reviewer.exe** во время работы — это остановит сервер. Приложение работает в связке: `.exe` запускает сервер, браузер отображает интерфейс.

#### Шаг 4. Настройте языковую модель

Без настроенной модели оценка работать не будет. Перейдите на вкладку **Настройки**:

1. В разделе **Модели** выберите нужную модель из списка.
2. Если рядом с моделью есть пометка «требует ключ» — нажмите **Изменить** и введите API-ключ.
3. Нажмите **Сохранить**.

Поддерживаемые провайдеры: OpenAI, Anthropic, Ollama (локальная). API-ключ хранится локально и никуда не передаётся кроме выбранного провайдера.

#### Шаг 5 (опционально). Установите Chromium для загрузки веб-страниц

Если вы планируете загружать документацию по URL, а не из файлов — потребуется браузер Chromium. Он не входит в `.exe` из-за большого размера (~300 МБ).

Для установки нужен Python 3.11 и playwright:

```bash
pip install playwright
playwright install chromium
```

После этого загрузка веб-страниц по URL заработает в разделе **По ссылке** на вкладке **Оценка**.

### Способ 2. Запуск из исходного кода (для разработчиков)

#### Требования

- **Python 3.11** — обязательно именно 3.11; PyMuPDF не совместим с Python 3.12+
- **Node.js** 18+

Проверьте, что Python 3.11 установлен:

```bash
py -3.11 --version
# Python 3.11.x
```

Если команда не найдена — скачайте Python 3.11 с [python.org](https://www.python.org/downloads/) (раздел «Looking for a specific release»).

#### 1. Клонировать репозиторий

```bash
git clone https://github.com/amihailov76/doc-reviewer.git
cd doc-reviewer
```

#### 2. Создать виртуальное окружение и установить зависимости

```bash
py -3.11 -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
```

> Виртуальное окружение изолирует зависимости проекта от системного Python. После активации `(venv)` появится в начале строки терминала. При следующих запусках достаточно выполнять только `venv\Scripts\activate`, а не пересоздавать окружение.

#### 2а. Установить браузер для парсинга веб-страниц (опционально)

Если планируете загружать документы по URL — установите Chromium. Выполняется один раз при активированном venv:

```bash
playwright install chromium
```

Chromium (~300 МБ) устанавливается глобально и не входит в `requirements.txt`. Без него вкладка **По ссылке** в разделе **Оценка** работать не будет.

#### 3. Запустить бэкенд

Убедитесь, что venv активировано (`(venv)` в начале строки), затем:

```bash
python run_dev.py
```

Бэкенд запустится на `http://localhost:8000`. Проверка: `http://localhost:8000/api/ping` → `{"status":"ok"}`

> **Предупреждения при запуске не являются стопперами.** При старте бэкенда могут появляться предупреждения вида `DeprecationWarning`, `UserWarning` от SQLAlchemy, pymorphy3 или других библиотек. Это ожидаемое поведение — сервер запустится корректно. Стоппером является только явная ошибка (`ERROR` / `Exception`) или отсутствие строки `Application startup complete` в выводе.

#### 4. Установить зависимости фронтенда и запустить

В **отдельном** терминале (venv здесь не нужен):

```bash
cd frontend
npm install
npm run dev
```

Vite запустит dev-сервер на `http://localhost:5173`.

> **Предупреждения при сборке фронтенда** (`warnings` от Vite/ESLint о неиспользуемых переменных, PropTypes и т.п.) не препятствуют работе приложения. Смотрите только на ошибки (`error`), которые прерывают сборку.

#### 5. Открыть в браузере

Перейдите на `http://localhost:5173` и настройте LLM-модель на вкладке **Настройки**.

Бэкенд (`python run_dev.py`) и фронтенд (`npm run dev`) должны работать одновременно в двух отдельных терминалах.

## Оценка веб-страниц

Помимо файлов, приложение поддерживает загрузку инструкций по URL. Страница загружается через headless-браузер Chromium (Playwright), что позволяет работать с JS-сайтами и SPA.

### Как использовать

1. На странице **Оценка** выберите вкладку **По ссылке**
2. Вставьте URL и нажмите **Загрузить**
3. После загрузки первой страницы можно добавить ещё — кнопка **+ Добавить страницу** появится в шапке документа
4. Все добавленные страницы оцениваются как один документ, каждая инструкция — отдельный раздел

### Особенности парсинга

Парсер оптимизирован для веб-справки Positive Technologies: распознаёт кастомные теги `<instruction>`, `<action>`, `<task>`, корректно нумерует шаги и убирает артефакты вёрстки (мягкие переносы `&shy;`). Каждый блок `<instruction>` на странице становится отдельным разделом в дереве структуры.

Для других сайтов используется универсальный fallback: HTML конвертируется в Markdown через `markdownify`.

### Установка зависимостей для веб-парсинга

```bash
py -3.11 -m pip install playwright beautifulsoup4 markdownify
py -3.11 -m playwright install chromium
```

## Конфигурация

### LLM-модели

Модели управляются через интерфейс (вкладка **Настройки → Модели**). При первом запуске загружаются дефолтные модели из `models.yml`.

Поддерживаемые провайдеры LLM:

| Провайдер | Base URL | Требует ключ |
|---|---|---|
| OpenAI | `https://api.openai.com/v1` | Да |
| Anthropic | `https://api.anthropic.com/v1` | Да |
| Ollama (локальная) | `http://localhost:11434/v1` | Нет |
| Любой OpenAI-совместимый | произвольный | По конфигурации |

API-ключ вводится в форме редактирования модели и хранится в локальной БД.

### Критерии оценки

Критерии управляются через интерфейс (вкладка **Настройки → Критерии оценки**). Дефолтные критерии при первом запуске загружаются из `criteria.md`.

Формат критериев в Markdown:

```markdown
## Группа критериев

### 1.1 Название критерия
Описание того, что проверяется.
```

При изменении `criteria.md` вручную — перезапустите бэкенд, изменения подтянутся автоматически. Файл `criteria.md`, расположенный рядом с `.exe`, имеет приоритет над встроенной версией.

## Сборка .exe

```bash
build.bat
```

Скрипт собирает фронтенд (`npm run build`), устанавливает зависимости Python, устанавливает Chromium через `playwright install chromium` и запускает PyInstaller. Результат: `dist\doc-reviewer.exe`.

При запуске `.exe` автоматически стартует бэкенд и открывает браузер. Данные хранятся в папке `data\` рядом с `.exe`.

> **Важно:** Chromium не упаковывается внутрь `.exe` — он устанавливается в системную папку Playwright (`%LOCALAPPDATA%\ms-playwright`) при сборке и должен присутствовать на машине пользователя. Для развёртывания на новой машине выполните `py -3.11 -m playwright install chromium` один раз.

## Этап 10: Проекты и контекст продукта (2026-04-05)

### Что изменилось

Добавлена концепция **Проекта** — контейнера для группы документов одного продукта с автогенерируемым контекстом.

### Новые модули

- `backend/routers/projects.py` — CRUD проектов, эндпойнт генерации контекста, управление документами проекта
- `backend/services/context_generator.py` — LLM-генерация контекста продукта из non-instruction разделов
- `frontend/src/pages/ProjectsPage.jsx` — список проектов
- `frontend/src/pages/ProjectPage.jsx` — детальная страница проекта

### Изменения в существующих модулях

**`backend/database.py`**
- Новая модель `Project` (поля: `id`, `name`, `product_context`, `context_generated_at`, `created_at`)
- Поле `project_id` в модели `Document` (FK → `projects.id`, nullable)
- Миграции `_migrate_add_project_column()` и `_migrate_create_default_project()`: при первом запуске создаётся проект «По умолчанию», все существующие документы переносятся в него

**`backend/services/evaluator.py`**
- `evaluate_instruction()` принимает новые параметры: `product_context`, `section_path`, `neighbor_titles`
- `_build_user_prompt()` включает блок контекста продукта и структурный контекст (путь раздела, заголовки соседних разделов) если они переданы

**`backend/routers/evaluation.py`**
- При оценке документа загружается `product_context` из проекта (если документ привязан)
- Для каждой инструкции вычисляются заголовки 2 предыдущих и 2 следующих разделов (`neighbor_titles`)
- Все эти данные передаются в `evaluate_instruction()`

**`backend/routers/documents.py`**
- `upload` и `replace` принимают опциональный `project_id` (query param)
- `/structure` возвращает `project_id` документа
- `_save_and_parse()` принимает `project_id`

**`frontend/src/App.jsx`**
- Добавлены маршруты `/projects` и `/projects/:id`
- Навигация: «📁 Проекты» теперь первый пункт, главная страница перенаправляет на `/projects`

**`frontend/src/pages/EvaluationPage.jsx`**
- Селектор проекта в шапке документа: выпадающий список всех проектов, смена проекта без перезагрузки
- Индикатор контекста: ✅ «контекст задан» / ⚠️ «контекст не задан» (кликабельный, ведёт в проект) / «контекст не используется»

### Новые API-эндпойнты

| Метод | Путь | Описание |
|---|---|---|
| GET | `/api/projects/` | Список проектов |
| POST | `/api/projects/` | Создать проект |

### Новые поля в БД

```sql
CREATE TABLE projects (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    product_context TEXT,
    context_generated_at DATETIME,
    created_at DATETIME
);

ALTER TABLE documents ADD COLUMN project_id INTEGER REFERENCES projects(id);
```

### Как работает генерация контекста

1. Берутся все `Instruction` с `classification = 'non-instruction'` из документов проекта (лимит 60 разделов)
2. Формируется промпт с этими разделами + системный промпт аналитика
3. LLM составляет описание продукта: название и класс, аудитория, 10–15 ключевых терминов, компоненты
4. Текст (~400–700 слов) сохраняется в `project.product_context`
5. При каждой оценке инструкций из этого проекта контекст подставляется в промпт

## Этап 11: Качество парсинга и улучшение промптов (2026-04-24)

### Что изменилось

#### Парсинг: жирное начертание и иконки

**`backend/services/parser/pdf_parser.py`** и **`backend/services/parser/docx_parser.py`**

Оба парсера теперь сохраняют семантику жирного начертания в тексте инструкций. Названия UI-элементов, предупреждения и термины, которые в документе выделены жирным, передаются в LLM обёрнутыми в `**...**`. Это позволяет критерию 2.2 (структура шагов) работать корректно, поскольку LLM видит форматирование наравне с содержимым.

Логика обработки в pdf_parser.py: каждый span обрабатывается на уровне фрагментов `(is_bold, text)`, одинаковые по bold-статусу фрагменты склеиваются с добавлением пробела при необходимости, разнородные фрагменты соединяются с той же проверкой на пробел. Заголовки разделов сохраняются без маркеров `**`.

Дополнительно исправлено определение иконок-шрифтов: если span после очистки PUA-символов пуст, но его ширина по bbox превышает 3 пт — значит, на странице есть видимый глиф из нестандартного шрифта. В этом случае вставляется метка `[иконка]`. Ранее глифы из не-PUA шрифтов пропадали без следа.

#### evaluator.py: архитектура промпта

**`backend/services/evaluator.py`**

Роль модели теперь извлекается динамически из `criteria.md` (раздел `## Роль`) и подставляется в SYSTEM_PROMPT перед задачей. Функции: `_extract_role()` и `_strip_role_section()`. Если `## Роль` в файле отсутствует — используется встроенная роль по умолчанию. Это позволяет менять роль эксперта через UI «Настройки → Критерии» без правки кода.

В системный промпт добавлены три директивных вопроса, которые LLM должен мысленно проработать перед оценкой каждой инструкции: понятность без контекста, неявные предусловия, нераскрытые объекты и процедуры. Вопросы работают как цепочка рассуждений и повышают чёткость выводов без изменения формата JSON-ответа.

Параметр `doc_type` удалён из `evaluate_instruction()` и `_build_user_prompt()` — он добавлял единственную строку в промпт и не влиял на логику оценки.

#### criteria.md: новая структура

Файл полностью переработан. Добавлена новая группа критериев **0 «Общее»** с критериями:

- **0.1 Заголовок** — проверяет, начинается ли заголовок с отглагольного существительного
- **0.2 Титульная фраза** — наличие краткой фразы, описывающей назначение инструкции

## Этап 12: Глоссарий продукта (YAKE) и умная переоценка при замене документа (difflib) (2026-04-24)

### Что изменилось

#### Глоссарий продукта — YAKE

**`backend/services/glossary.py`** (новый)

При каждой загрузке документа автоматически извлекается список устоявшихся терминов продукта. Используется YAKE (статистический экстрактор ключевых фраз) поверх текста всех разделов; при наличии pymorphy3 текст предварительно лемматизируется для корректной дедупликации форм. Результат фильтруется: убираются глагольные формы, стоп-слова, цифровые строки, дубли. Итог — не более 30 терминов сохраняется в `Document.glossary` (JSON).

Глоссарий подставляется в каждый LLM-запрос блоком «ТЕРМИНЫ ПРОДУКТА» и сообщает модели, какие термины являются устоявшимися для данного продукта и не требуют расшифровки в каждом разделе. Снижает количество ложных срабатываний по критериям 1.1 (цель и контекст) и 1.2 (предварительные условия).

**`backend/routers/documents.py`**

После парсинга разделов вызывается `extract_glossary(result.sections)`, результат записывается в `doc.glossary`. Ошибка при извлечении глоссария не прерывает загрузку документа — записывается пустой список.

**`backend/services/evaluator.py`**

Функция `evaluate_instruction()` и `_build_user_prompt()` принимают параметр `glossary: list[str] = None`. Вспомогательная функция `glossary_to_prompt_block()` формирует текстовый блок для промпта.

**`backend/routers/evaluation.py`**

`doc.glossary` загружается из БД и передаётся в `_call_llm_with_heartbeat()` и `evaluate_single()`.

#### Умная переоценка при замене документа — difflib

**`backend/services/differ.py`** (новый)

Сервис сопоставления версий документа. Функция `match_sections(old, new)` для каждого раздела новой версии ищет пару в старой — сначала по точному совпадению заголовка, затем по `SequenceMatcher` с порогом 0.70. Для найденных пар вычисляется схожесть содержимого:

| Ratio | Действие |
|---|---|
| ≥ 0.95 | `copy` — раздел не изменился, оценка переносится без LLM-вызова |
| 0.60–0.95 | `hint` — раздел изменился частично, генерируется `diff_hint` |
| < 0.60 или нет пары | `fresh` — оценивается с нуля |

`diff_hint` — краткое описание изменений: количество добавленных/удалённых строк и примеры изменённых фрагментов (до 2 штук из `unified_diff`).

**`backend/routers/documents.py`**

`replace_document()` переработан: до удаления старого документа снимает снимок его инструкций вместе с оценками. После парсинга новой версии вызывает `_apply_diff_matches()`: для разделов с `action=copy` создаёт `Evaluation` напрямую в БД (без LLM), для `action=hint` записывает `diff_hint` в поле `Instruction.diff_hint`.

**`backend/services/evaluator.py`**

Параметр `diff_hint: str = None` в `evaluate_instruction()` и `_build_user_prompt()`. Если задан — добавляется блок «ИЗМЕНЕНИЯ В РАЗДЕЛЕ» с описанием что изменилось и указанием оценивать актуальный текст.

**`backend/routers/evaluation.py`**

`instr.diff_hint` передаётся в оба вызова `evaluate_instruction()`.

### Новые поля в БД

```sql
ALTER TABLE documents    ADD COLUMN glossary   JSON;
ALTER TABLE instructions ADD COLUMN diff_hint  TEXT;
```

Обе колонки добавляются автоматически при старте через `_migrate_add_glossary_column()` и `_migrate_add_diff_hint_column()`.

## Известные ограничения

- **Python 3.11** — PyMuPDF несовместим с Python 3.12+
- **Иконки в PDF** — символы из символьных шрифтов (Private Use Area), реализованные как графика, не извлекаются PyMuPDF. В системном промпте LLM явно указано игнорировать такие артефакты при оценке
- **Только локальный запуск** — приложение рассчитано на одного пользователя, многопользовательский режим не поддерживается
- **Парсинг веб-страниц** — оптимизировано для структуры веб-справки Positive Technologies (`<instruction>`, `<action>`, `<task>`). Для других сайтов используется универсальный fallback через markdownify — качество извлечения текста может быть ниже
- **Chromium** — не входит в `.exe`, требует отдельной установки через `playwright install chromium`
