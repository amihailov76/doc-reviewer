# CLAUDE.md — doc-reviewer

## Что это за проект

Десктопное приложение для оценки полноты инструкций в технической документации. Целевая аудитория — технические писатели, документирующие сложные ИБ-продукты. Загружает документы (PDF, DOCX, MD, TXT), классифицирует разделы как инструкции, оценивает каждую инструкцию через LLM по критериям из `criteria.md`, выдаёт цветовой результат и рекомендации.

## Стек

- **Backend:** Python, FastAPI, SQLAlchemy, SQLite
- **Frontend:** React 18 + Vite (SPA), раздаётся как статика из FastAPI
- **Сборка:** PyInstaller → один `.exe` для Windows
- **LLM:** OpenAI-совместимые API + Anthropic (настраивается через UI)

## Архитектурные ограничения — не менять без обсуждения

**SQLite без Alembic.** Миграции написаны вручную в `database.py` (`init_db`, `_migrate_*`). Alembic и другие migration-инструменты несовместимы со сборкой в exe и не нужны при текущей схеме. При добавлении новых колонок дописывать `ALTER TABLE` в `init_db` вручную.

**PyInstaller-совместимость.** Все пути к файлам определяются через проверку `sys.frozen` и `sys._MEIPASS`. Не использовать `__file__` напрямую в сервисах — только через вспомогательные функции вида `_find_criteria()`. Новые файлы данных (не код) добавлять в `.spec`-файл.

**Синхронный httpx в роутерах.** LLM-запросы идут через синхронный `httpx.Client` внутри `def`-роутеров FastAPI (не `async def`). Это намеренно — позволяет использовать `threading` для SSE с heartbeat. Не переводить на async без необходимости.

**Фронтенд без внешних UI-библиотек.** Только React + react-router-dom. Не добавлять MUI, Ant Design и т.п. — это увеличивает бандл и усложняет сборку.

## Инварианты — не нарушать

- **API-ключи не отдавать на фронтенд.** `ModelConfig.dict()` намеренно не включает `api_key`. Не добавлять его в сериализаторы.
- **`overrides` не сбрасывать при переоценке.** Пользователь вручную отмечает ложные срабатывания. При обновлении `Evaluation` обновлять только `color`, `criteria_results`, `recommendations`, `model_used`, `evaluated_at`.
- **Классификация `non-instruction` не уходит на оценку LLM.** В `evaluation.py` фильтр: `classification.in_(["instruction", "possible"])`. Не убирать.
- **`criteria.md` рядом с `.exe` имеет приоритет.** Пользователь может положить свой файл рядом с приложением. Порядок поиска: рядом с exe → `_MEIPASS` → корень проекта.

## Структура проекта

```
backend/
  main.py              — FastAPI app, подключение роутеров, раздача SPA
  database.py          — SQLAlchemy модели, init_db, миграции при старте
  config.py            — работа с моделями LLM (загрузка, активация)
  routers/
    documents.py       — загрузка, парсинг, структура документов
    evaluation.py      — оценка через LLM, SSE-стрим прогресса
    instructions.py    — управление инструкциями (классификация, include)
    snapshots.py       — слепки результатов оценки
    groups.py          — группы продуктов для сравнения снепшотов
    config_router.py   — управление моделями LLM и наборами критериев
    web.py             — загрузка документов по URL (Playwright)
  services/
    detector.py        — классификация разделов по 3 признакам (pymorphy3)
    evaluator.py       — запрос к LLM, retry-логика, парсинг ответа
    parser/            — парсеры: pdf (PyMuPDF), docx, md/txt

frontend/src/
  pages/               — страницы: DocumentPage, EvaluationPage, ResultsPage,
                         SettingsPage, SnapshotPage, SnapshotsPage
  components/          — DocumentTree и другие компоненты
```

## Модели БД

- `Document` → `Instruction` (1:N, cascade delete)
- `Instruction` → `Evaluation` (1:1, cascade delete)
- `ProductGroup` → `Snapshot` (1:N, cascade delete)
- `Model` — конфигурации LLM
- `CriteriaSet` — наборы критериев оценки (Markdown)

## Язык и стиль кода

- Комментарии и строки документации — **на русском**.
- Сообщения об ошибках в `EvaluationError` — на русском, адресованы пользователю.
- Логи и технические `detail` — на русском.
- Имена переменных, функций, классов — на английском.
- Новые роутеры добавлять в `backend/routers/`, подключать в `main.py`.
- Новые сервисы добавлять в `backend/services/`.

## Что запланировано (не реализовано)

### Концепция проекта (Project)

Приоритет — следующий этап разработки.

Цель: группировать документы одного продукта, хранить контекст продукта для улучшения точности LLM-оценок.

Схема:
```
Project
├── id, name
├── product_context   — текст (~500 токенов): название, аудитория, глоссарий, архитектура
├── context_generated_at
└── documents[]       — связанные Document
```

Поведение:
- Контекст генерируется автоматически из «вводных» разделов (`non-instruction`) через LLM-запрос при первом создании проекта.
- Пользователь может отредактировать контекст вручную.
- Кнопка «Перегенерировать» — пересоздаёт из текущего состава документов.
- `product_context` подставляется в промпт `evaluator.py` при каждой оценке.
- Один пользователь ведёт несколько проектов (разные продукты).

### Улучшение контекста оценки

После реализации проектов — дополнительный контекст в промпте:
- `section_path` инструкции (уже есть в БД, не передаётся в LLM).
- Заголовки 2–3 соседних разделов.

### RAG (долгосрочно)

Векторный поиск по всему комплекту документов для поиска определений терминов и связанных процедур. Рассматривать `sqlite-vec` (без отдельной векторной БД).

## Документация — обязательно по завершении каждого этапа

После реализации каждого значимого этапа (новая фича, изменение архитектуры, исправление поведения) обновлять два файла.

### `docs/README.md` — техническая документация

Фиксирует всё, что важно разработчику: что изменилось в архитектуре, какие модули затронуты, новые таблицы или поля БД, новые эндпойнты API, изменения в промптах или логике классификации. Формат — свободный технический текст, структурированный по разделам.

При завершении нового этапа:
- Добавить отметку `[x]` в раздел **Статус этапов**.
- Дополнить разделы **Возможности**, **Структура проекта**, **База данных**, **Известные ограничения** — если изменения затрагивают эти темы.
- Если появляются новые команды или зависимости — обновить **Быстрый старт** и **Конфигурацию**.

### `docs/doc-reviewer-guide.pdf` — пользовательское руководство

Описывает изменения на уровне пользовательских сценариев: что появилось в UI, как теперь работает тот или иной экран, какие новые действия доступны пользователю. Без технических деталей реализации — только то, что видит и делает технический писатель. При обновлении сохранять существующую структуру документа, добавляя новые разделы или дополняя существующие.

**Перед обновлением руководства обязательно прочитать `docs/styleguide.md`** и соблюдать все правила оформления и стиля, принятые в команде. Стайлгайд будет пополняться — всегда читать его актуальную версию.

## Правила работы с изменениями

При любых изменениях файлов всегда явно указывать, требуется ли перезапуск бэкенда и/или фронтенда:

- **`.py` файлы** (backend/) — требуют перезапуска бэкенда (`python run_dev.py`)
- **`.js` / `.jsx` / `.ts` / `.tsx` / `.css` файлы** (frontend/src/) — Vite подхватывает автоматически, перезапуск не нужен
- **`criteria.md`** — не требует перезапуска; новое содержимое подтягивается через UI (Настройки → Критерии). Если БД пуста — только тогда читается при старте
- **`models.yml` / конфиги** — требуют перезапуска бэкенда
- **`.spec` файл / `build.bat`** — только для сборки exe, на dev-режим не влияют

Если изменение затрагивает парсеры (pdf_parser, docx_parser) — дополнительно указывать, что для применения к уже загруженным документам нужна их переразборка.

## Известные технические долги

- `datetime.utcnow()` — deprecated в Python 3.12+, заменить на `datetime.now(timezone.utc)` когда будет время.
- `_find_criteria()` дублируется в `evaluator.py` и `database.py` — вынести в утилиту.
- API-ключи хранятся в SQLite в открытом виде — приемлемо для десктопа, но стоит зашифровать через `cryptography` (зависимость уже есть в `requirements.txt`).
